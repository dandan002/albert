# Domain Pitfalls: Prediction Market Trading Systems

**Domain:** Automated prediction market trading (Polymarket, Kalshi)
**Researched:** 2026-04-12
**Overall confidence:** HIGH

---

## Critical Pitfalls

Mistakes that cause rewrites, significant losses, or complete system failure.

### Pitfall 1: Polymarket ECDSA Authentication Not Implemented

**What goes wrong:** The `PolymarketAdapter` cannot execute orders in production. The adapter lacks the per-request ECDSA signing required by the Polymarket CLOB API. Orders will fail with authentication errors.

**Why it happens:** Only `POLY_ADDRESS` and `POLY_API_KEY` headers are sent. Production requires cryptographic signing of each request with the private key.

**Consequences:** Polymarket orders silently fail or return 401 errors. Strategies generate order intents that never execute.

**Prevention:** Implement L2 ECDSA request signing per Polymarket CLOB docs before enabling PolymarketAdapter in production. Document the required env vars: `POLY_ADDRESS`, `POLY_API_KEY`, `POLY_API_SECRET`, `POLY_PASSPHRASE`.

**Warning signs:** API responses return 401; adapter logs show "authentication failed" errors.

**Phase:** This is a prerequisite — implement in phase before any Polymarket trading.

---

### Pitfall 2: PolymarketAdapter Not Registered in Main Entry Point

**What goes wrong:** `PolymarketAdapter` is imported but never instantiated in `albert/__main__.py`. Even if authentication works, orders cannot be executed.

**Why it happens:** Line 62 only creates `"kalshi": KalshiAdapter()`. PolymarketAdapter exists but is not added to the adapters dict.

**Consequences:** Polymarket order intents are generated by strategies but silently dropped with "no_adapter" log message. No error surfaces to user.

**Prevention:** Conditionally add `PolymarketAdapter` to the adapters dict when Polymarket env vars are present and auth is implemented.

**Warning signs:** Strategy emits Polymarket orders; execution logs show "no_adapter" for polymarket markets.

**Phase:** Same as Pitfall 1 — fix adapter registration before enabling Polymarket trading.

### Pitfall 3: Position Sizing Without Edge Calculation

**What goes wrong:** Strategies emit order intents without calculating whether the price provides a genuine edge. Kelly criterion sizing assumes edge > 0. If strategies trade at market price without edge, Kelly fraction amplifies losses.

**Why it happens:** The strategy interface's `estimate_edge()` method may return arbitrary values. Without proper probability estimates compared to market price, sizing is mathematically wrong.

**Consequences:** Aggressive position sizing on trades with no edge rapidly depletes bankroll. Kelly criterion amplifies negative expected value.

**Prevention:** Require `estimate_edge()` to compare strategy's probability estimate against current market price. Only emit order intents when edge exceeds configurable threshold (e.g., `min_edge: 0.05`).

**Warning signs:** High trade frequency with low or negative realized P&L; win rate below 50% but position sizes increasing.

**Phase:** Strategy development phases — ensure all strategies compute edge before emitting orders.

---

### Pitfall 4: Ignoring Liquidity in Thin Markets

**What goes wrong:** The system enters positions in markets with insufficient volume. When exiting, slippage destroys any theoretical edge.

**Why it happens:** No liquidity check before order execution. Strategy may emit order intent for any market regardless of orderbook depth.

**Consequences:** Enter at reasonable price, exit at 50-70% of mark due to lack of counterparties. A 10% edge becomes a 30% loss after slippage.

**Prevention:** Add a pre-trade liquidity check: verify orderbook depth at current price supports the intended notional. Reject orders if depth < 2x position size or total volume < $5,000.

**Warning signs:** Orders fill but exit at significantly worse prices; unrealized P&L diverges from expected based on entry price.

**Phase:** Execution engine improvements — add liquidity validation to risk checks.

---

### Pitfall 5: No Graceful Shutdown Causes Data Loss

**What goes wrong:** The system runs with `asyncio.gather()` and no signal handling. Ctrl+C or SIGTERM causes abrupt termination. Pending fills may not be persisted, position state becomes inconsistent.

**Why it happens:** `__main__.py` has no signal handlers or graceful cancellation scopes.

**Consequences:** After restart, positions in database do not match actual exchange positions. Missing fill records break P&L tracking. Corrupted state requires manual reconciliation.

**Prevention:** Add `signal.signal()` handlers for SIGINT/SIGTERM. Use `asyncio.TaskGroup()` with cancellation scopes to ensure `_persist_fill()` and connection cleanup run before exit.

**Warning signs:** After restart, positions table has mismatched contract counts vs. exchange; missing fills in history; daily P&L jumps.

**Phase:** Infrastructure/resilience phase — add before production deployment.

## Moderate Pitfalls

Mistakes that cause degraded performance, edge erosion, or operational issues.

### Pitfall 6: Fee Impact Underestimation

**What goes wrong:** Strategies calculate expected value without accounting for exchange fees, spreads, and withdrawal costs. A 5% theoretical edge becomes -2% after costs.

**Why it happens:** Execution engine computes Kelly sizing but doesn't adjust for fees in edge calculations. Different exchanges have different fee structures (Polymarket: 0-0.2%, Kalshi: 5% on wins).

**Consequences:** Net negative expected value trades appear profitable. Consistent losses over time despite correct directional predictions.

**Prevention:** Subtract fees from edge before sizing. For each exchange, calculate net edge: `gross_edge - fee_rate * win_probability`. Only emit orders when net edge exceeds minimum threshold.

**Warning signs:** Win rate > 50% but overall P&L negative; better win rate correlates with worse returns.

**Phase:** Strategy refinement — ensure edge calculations include fees.

---

### Pitfall 7: EventBus Has No Backpressure

**What goes wrong:** `EventBus` uses unbounded `asyncio.Queue` for each subscriber. During market surges, queues grow without limit, consuming memory.

**Why it happens:** Queues created without `maxsize` parameter in `albert/events.py`.

**Consequences:** OOM crash during high-frequency market data events. System becomes unresponsive, stops processing orders.

**Prevention:** Use `asyncio.Queue(maxsize=N)` with configurable limit. Decide on backpressure strategy: drop oldest, block publisher, or emit alert.

**Warning signs:** Memory usage growing unbounded; queue depth metrics spike; processing latency increases.

**Phase:** Infrastructure phase — add before handling high-volume markets.

---

### Pitfall 8: Broad Exception Handling Swallows Errors

**What goes wrong:** Strategy engine catches `Exception` broadly (lines 48, 72 in `engine.py`). Errors are only logged — no metrics, no health alerts, no disable mechanism.

**Why it happens:** Generic exception handling assumes all errors are retriable and recoverable.

**Consequences:** Broken strategies silently continue receiving events. Errors compound; system appears healthy but strategies are not functioning. No visibility into actual system health.

**Prevention:** Differentiate retriable vs. fatal errors. After N consecutive failures, disable the strategy and emit `StrategyHaltedEvent`. Add structured error metrics.

**Warning signs:** Log shows repeated error messages for same strategy; positions not updating; orders stopping unexpectedly.

**Phase:** Error handling improvements — add strategy health monitoring.

---

### Pitfall 9: No Order Status Tracking

**What goes wrong:** System places orders and records them as fills immediately, but never checks if the order was actually filled. Partial fills or rejections go undetected.

**Why it happens:** No background task monitoring open orders. `cancel_order()` exists but is never called.

**Consequences:** Position records don't match actual exchange state. If order is partially filled or rejected, system continues as if full fill occurred. P&L diverges from reality.

**Prevention:** Add a background task that polls open orders every N seconds. Reconcile fill status with exchange response. Update positions accordingly.

**Warning signs:** Position contract count doesn't match exchange; fills table shows more contracts than actually held; P&L drift from expected.

**Phase:** Execution engine improvements — add order status polling.

---

### Pitfall 10: SQLite Write Contention Under Load

**What goes wrong:** SQLite supports only one writer at a time. Multiple modules write simultaneously (`ExecutionEngine`, `PortfolioTracker`, `StrategyEngine`, `_ttl_cleanup`).

**Why it happens:** No write batching; every event triggers immediate DB write. WAL mode helps but doesn't eliminate contention.

**Consequences:** `database is locked` errors; latency spikes during high trade volume; periodic slowdowns during TTL cleanup.

**Prevention:** Batch writes within a time window (e.g., collect writes, flush every 1-2 seconds). Use separate connection for TTL cleanup. Consider PostgreSQL for multi-writer support.

**Warning signs:** "database is locked" errors in logs; periodic latency spikes; execution engine slower during high activity.

**Phase:** Scalability phase — optimize DB write patterns before high-volume trading.

## Minor Pitfalls

Mistakes that cause confusion, debugging difficulty, or minor operational issues.

### Pitfall 11: Dynamic Strategy Loading Has No Validation

**What goes wrong:** `importlib.import_module()` loads strategy classes from DB. A typo in `class_path` silently prevents strategy activation with only a log message.

**Why it happens:** `class_path` stored in DB with no validation at insert time. Error handling is broad `except Exception` that only logs.

**Consequences:** Strategy appears enabled but never runs. No alert that activation failed. Debugging requires reading logs to discover loading failure.

**Prevention:** Add `validate_class_path()` function that dry-runs the import when strategy is inserted. Validate on startup before marking enabled. Consider a strategy registry pattern.

**Warning signs:** Strategy enabled but no market data events processed; no orders emitted; log shows import failure.

**Phase:** Strategy engine improvements — add validation.

---

### Pitfall 12: Database Path is Hardcoded Relative

**What goes wrong:** `DB_PATH = Path("albert.db")` is relative to working directory. Running from different directory uses different database file.

**Why it happens:** No configuration for database path; assumes always run from project root.

**Consequences:** Split state — positions exist in one DB, strategy configs in another. Data inconsistent depending on how system is started.

**Prevention:** Make `DB_PATH` configurable via environment variable `ALBERT_DB_PATH` or `config.json`.

**Warning signs:** Running `python -m albert status` from different directory shows different state; config changes don't persist.

**Phase:** Configuration improvements — make paths configurable.
